# Deterministic read-only bundle pattern

Plant Diversity commits trimmed `.rds` artifacts beside the app so core exploration is fast, reproducible, and independent of a live upstream request. The files are a versioned read-only data store; they are not an informal cache.

## When to use it

Use committed bundles when data are read-only, update on a release cadence, fit comfortably in the deployment, and are expensive or fragile to fetch at runtime. Use a database or warehouse when users write data, the dataset is too large, freshness is operationally critical, or cross-language/query requirements dominate.

`.rds` preserves R types and attributes and is fast to read. Prefer Parquet when non-R consumers need the same analytical table. Compression choice is part of the deterministic build contract.

## Fetch boundary

Fetch into an empty staging root with:

- explicit product, query cutoff and immutable query/snapshot ID, site/entity inventory, and source version;
- an official upstream release only when that release was actually selected and recorded;
- source/license/query receipts;
- failure collection that exits non-zero;
- no write credential in the validation job;
- no reuse of unknown partial downloads.

Resumability may help a developer fetch, but a candidate is not complete until every expected entity is present in the same staged release. Never fill a failed new fetch with an old production file.

## Build boundary

Build from immutable staged inputs into a separate candidate root. Trim only after documenting the source-to-bundle mapping. Preserve the fields required to reconstruct classifications, denominators, support, joins, and release provenance. Record the actual build date separately from source cutoff/snapshot identity, and preserve raw/source digests plus the builder commit.

For Plant Diversity, the candidate contains exactly:

- 46 `data/sites/<SITE>.rds` plant bundles;
- 46 `data/env/<SITE>.rds` context bundles;
- `data/site_index.rds` and `data/search_index.rds`;
- the SRER default artifact;
- expected/reference, authority, and completeness artifacts;
- explicit source/build receipts, with an official release value only when that release was selected.

Every expected bundle and its index must carry one complete matching source receipt. Partial or disagreeing receipt fields fail closed. Build twice in isolated roots with pinned R/package/numeric settings and compare exact bytes. A wall-clock timestamp may appear only when supplied as an explicit candidate input; it must not silently make otherwise identical builds differ.

Legacy families with missing upstream receipts may be frozen by a content-addressed repository receipt and used for explicitly descriptive exact-byte results. Their missing build date, release, cutoff, query ID, and raw-source digest remain unknown; file mtimes, repository dates, manifests, and runtime hashes cannot repair those gaps.

## Runtime boundary

The production app reads committed bundles only:

```r
load_bundle <- function(site) {
  path <- file.path("data", "sites", paste0(site, ".rds"))
  if (!file.exists(path)) return(NULL)
  tryCatch(readRDS(path), error = function(error) NULL)
}
```

Missing/corrupt data produces a visible, recoverable error. Production does not fall through to a live NEON fetch: that would change source vintage, schema, latency, failure behavior, and provenance inside a published release.

Filter and derive in memory only under registered observation/estimator contracts. The source bundle, analysis snapshot, plot summary, PDF, and exports must agree.

## Manifest and promotion boundary

The deployment manifest is an exact closure over runtime files, checksums, package identities, repositories, and source URLs. After any code, data, or vendored-asset change:

1. regenerate derived artifacts;
2. regenerate `manifest.json` under the pinned release environment;
3. run the full bundle/manifest verifier;
4. commit the exact validated bytes to a review branch;
5. merge only after CI proves the committed and regenerated bytes are identical;
6. publish that reviewed commit and verify it in Connect and the public browser.

Do not infer deployment from a Git push. Record the Connect revision/deployment identity and semantic health receipt.

## Refresh publication

Scheduled refreshes open or update a review PR. Review inventory, schema, row-count, vintage, deletion, metric, and bundle-size changes before promotion. A refresh never pushes directly to `master` and never overwrites the last known-good bundle set without a rollback path. A skip-download run revalidates the existing bytes and receipt without stamping new source metadata.

## Minimum bundle receipt

Every promoted bundle family records:

- source product and license;
- actual build date, query cutoff and immutable query/snapshot ID;
- true selected official release when applicable;
- expected and actual inventory;
- schema/version;
- raw/source and derived per-file and aggregate hashes;
- builder commit;
- observation/estimator contract version;
- missingness/opportunity semantics;
- build environment;
- reviewed commit and deployment identity.
