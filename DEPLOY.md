# Deploy & hosting runbook

This app has **always been git-backed Posit Connect Cloud** — it never had a
shinyapps.io/rsconnect tree, no `scripts/deploy.R`, no shinyapps secrets. Connect Cloud
deploys it straight from the public GitHub repo and **auto-republishes on every push to
`master`**. A GitHub-Pages landing page sits in front (an og card + a cold-start pre-warm),
mirroring the fleet convention (Small Mammal Tracker, WaterChem Analyte Viewer).

## The shape of the move

```
  Posit Connect Cloud (runs the app)          GitHub Pages  (docs/)
  └─ runs global/ui/server + R/ + www/        └─ landing page → "Launch" → app URL
     + data/site_index.rds (the picker)          + og:image social card
     + data/sites/*.rds  (46 site bundles)        + fetch() pre-warm ping (cold-start fix)
     + data/env/*.rds    (46 env overlays)
     + data-sample/demo.rds  (SRER demo)
        from the PUBLIC repo, branch master,
        via manifest.json — BUNDLE-ONLY
```

The landing page lives at [`docs/index.html`](docs/index.html) and the social card at
[`docs/og-image.png`](docs/og-image.png) (regenerate with `Rscript scripts/make_og_image.R`).
After you deploy the app, set the app URL in **one place** — the `APP_URL` constant near the
bottom of `docs/index.html` — then publish Pages. Until the real URL is known, `APP_URL` holds
the placeholder sentinel `REPLACE-WITH-CONNECT-CLOUD-URL` (the launch script detects it, the
same way the mammal template handles its `REPLACE-WITH...` sentinel).

## Option A — Posit Connect Cloud (recommended; in use today)

Connect Cloud deploys an R app straight from a **public** GitHub repo using `manifest.json`
(already generated by [`scripts/write_manifest.R`](scripts/write_manifest.R)).

The manifest is **lean and bundle-only**: it lists `global.R` + `ui.R` + `server.R`, `R/`,
`www/`, the `data/site_index.rds` picker index, the 46 `data/sites/*.rds` site bundles, the
46 `data/env/*.rds` environment overlays, and the `data-sample/demo.rds` SRER demo — and
**91 CRAN packages, NOT `neonUtilities`.** `neonUtilities` is referenced by a *computed* name
in `global.R` (`.NEON_PKG <- paste0("neon", "Utilities")`) precisely so the rsconnect
dependency scanner never pins it: it has no wasm build (which would block a future shinylive
export) and a live NEON pull on a cold free-tier worker is a hang risk. The deployed app runs
**entirely from the bundled `.rds` files**; the optional live-fetch path
(`LIVE_FETCH`, gated on `Sys.getenv("PDE_LIVE")`) only activates where `neonUtilities` is
actually installed (local dev). Force bundle-only anywhere with **`PDE_LIVE=0`** (it's already
the default).

1. Sign in at <https://connect.posit.cloud> with GitHub.
2. **New Content → Shiny → from Git**, pick `tgilbert14/NEON-Plant-Diversity`, branch
   **`master`**. Connect reads `manifest.json` (`appmode: shiny`) and auto-detects the
   multi-file Shiny app (`global.R` + `ui.R` + `server.R`) — there is **no single "primary
   file"** to choose for a `ui.R`/`server.R` app; `global.R` is sourced automatically.
3. It reads `manifest.json`, restores the 91 packages, and serves. **Auto-republishes on
   every push to `master`.**
4. Copy the published URL → set `APP_URL` in `docs/index.html` (and the README badge),
   replacing the `REPLACE-WITH-CONNECT-CLOUD-URL` sentinel.

Regenerate the manifest whenever runtime dependencies change:
```sh
# R with the app's runtime packages — R 4.3.1 has them all:
Rscript scripts/write_manifest.R     # writes manifest.json + self-checks neonUtilities is absent
```
The script prints a package count and an explicit
`OK: neonUtilities is NOT in the manifest` line — if it ever warns that neonUtilities leaked,
the anti-pin broke; do not deploy until it's clean again.

**Cold start:** the free tier sleeps after idle. The landing page's pre-warm `fetch(APP_URL)`
fires on load and wakes the worker while the visitor reads, so the app is usually warm by the
time they click **Launch**.

## Option B — Shinylive / WebAssembly (best long-term: static, zero server)

Because the deployed app has **no hard runtime dependency on `neonUtilities`** (read-only
bundled `.rds`, the live path is dev-only), it's a genuine shinylive candidate — it could
serve from GitHub Pages alongside the landing page, with **zero cold start**. Two gates,
both must pass:

- **Per-package wasm binaries.** Every package needs a wasm build at
  <https://repo.r-wasm.org/>. **Verify each before committing** — one missing binary kills the
  static build: `leaflet` (the picker map), `plotly`, `DT`, `bslib`, `bsicons`, `shinyjs`,
  `shinycssloaders`, `dplyr` / `tidyr` / `stringr`, `RColorBrewer`, `htmltools` (plus
  `shiny`, `tibble`). These are mainstream and usually present — but check per-package.
- **Bundle size shipped to the browser.** Unlike Connect Cloud (where the bundle sits on the
  server), shinylive ships the whole dataset to each visitor: the **46 site `.rds` + the 46
  env overlays + `data/site_index.rds` + the SRER demo**. Confirm the total wasm payload is
  acceptable before promising this path — this dataset is meaningfully larger than the mammal
  app's.

```r
# install.packages("shinylive")
shinylive::export(".", "docs/app")   # emits a static site under docs/app/
# then APP_URL = "app/"  (relative) — ONE GitHub-Pages deploy serves landing + app
```

If both gates are clean, this is the cheapest home and removes Connect Cloud's cold start. If
either fails, stay on Connect Cloud (Option A).

## Automatic data refresh

`.github/workflows/refresh-data.yml` rebuilds the bundles on a schedule and **pushes the
refreshed `data/` to `master`**. Because this app is git-backed Connect Cloud, **that push is
the deploy** — Connect Cloud auto-redeploys from `master` with no further action. There are
**no shinyapps secrets and no `deploy.R`** here: this app never had a shinyapps/rsconnect
tree, so unlike the mammal app's workflow there is no separate deploy step to wire up.

The refresh sequence the workflow runs (and the order matters):

1. **Rebuild the bundles** — `Rscript scripts/fetch_plant_all.R` (resumable, chunkable NEON
   pull on R 4.3.1) → `Rscript scripts/bundle_plant_data.R` (trims raw dumps →
   `data/sites/*.rds` + rebuilt `data/site_index.rds` + demo), plus the env overlays.
2. **Regenerate the manifest** — `Rscript scripts/write_manifest.R`. Connect Cloud pins a
   **checksum per bundled file** in `manifest.json`; a changed `.rds` whose checksum wasn't
   refreshed will silently keep serving stale data (or fail the deploy). **Never skip this
   after a data rebuild.**
3. **Commit + push to `master`** — `git add data/ manifest.json && commit && push`. The push
   triggers the Connect Cloud auto-redeploy; the live app updates with no manual republish.

The easy steps to forget are **2 and 3** — a local rebuild changes nothing in production until
the manifest checksums match and the push lands. (This bit a sibling app: bundles looked
updated locally, the live app didn't change.) Keep that note in the refresh script's header.

**Optional `NEON_TOKEN` repo secret** (Settings → Secrets and variables → Actions): a NEON API
token raises the pull rate limit so the CI fetch finishes faster. It's optional — the fetch
works token-less, just slower — and the deployed app never needs it (bundle-only).
